## Вставка в Hadoop/Impala и PostgreSQL

Предположим, что у нас есть идеальный `DataFrame`:

- отсутствуют пропущенные значения (`NULL`, `NaN`, `None`);
- типы данных `DataFrame` полностью соответствуют типам столбцов целевой таблицы;
- названия столбцов совпадают с названиями в таблице.

В этом случае вставка данных может выполняться по одному универсальному шаблону:

```
DataFrame
    │
    ▼
Формирование SQL INSERT
    │
    ▼
Выполнение INSERT
```

То есть логика загрузки практически одинакова как для `PostgreSQL`, так и для `Hadoop/Impala`. Отличаться будет только драйвер подключения и способ выполнения SQL-запроса.

**После этого универсальный шаблон постепенно дополняется проверками**

На практике DataFrame редко бывает идеальным, поэтому перед вставкой обычно выполняются дополнительные проверки и преобразования.

| Проверка                               | Для чего нужна                                           |
| -------------------------------------- | -------------------------------------------------------- |
| Проверка подключения к БД              | Убедиться, что соединение успешно установлено            |
| Проверка существования схемы и таблицы | Исключить ошибку при вставке                             |
| Проверка совпадения списка столбцов    | DataFrame и таблица должны иметь одинаковую структуру    |
| Проверка типов данных                  | Типы DataFrame должны соответствовать типам столбцов БД  |
| Преобразование типов                   | Например, `numpy.int64 → int`, `numpy.float64 → float`   |
| Обработка пропусков                    | Замена `NaN`, `None`, `pd.NA` на `NULL`                  |
| Экранирование строк                    | Корректная обработка кавычек и специальных символов      |
| Batch-вставка                          | Загрузка данных частями для повышения производительности |
| Обработка ошибок                       | `try/except`, `rollback`, логирование ошибок             |
|Проверка количества строк               |Если DataFrame пуст (df.empty), загрузку можно пропустить, не открывая соединение с БД|

**Особенность PostgreSQL**

При работе с PostgreSQL большинство этих задач уже решает драйвер `psycopg2`, особенно если использовать `execute_values()`. Он автоматически:

- преобразует Python-тип `None` в `NULL`;
- экранирует строки;
- корректно сериализует даты и время;
- формирует безопасный SQL-запрос.

**Особенность Hadoop/Impala**

При ручной генерации SQL (как в моей функции) многие преобразования приходится выполнять самостоятельно:

- преобразовывать `NaN` в `NULL`;
- приводить `numpy.int64` и `numpy.float64` к стандартным типам Python (int, float);
- экранировать строки;
- следить за корректным синтаксисом SQL.

Поэтому универсальная функция для Impala обычно содержит больше собственной логики подготовки данных, чем аналогичная функция для PostgreSQL.
